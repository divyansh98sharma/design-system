  <div className="ph-principle">

  <div className="ph-principle">

  </div>
  <div className="ph-principle">

## Why a design system?

<p>
  eClinicalWorks products span dozens of screens built across multiple teams over many years.
  Without a shared language, each team inevitably makes slightly different decisions about
  spacing, colour, interactive states, and motion — and patients and clinicians end up with
  a fragmented experience.
</p>

<p>
  The eCW Design System solves this by providing a <strong>single authoritative library</strong>
  of primitives that every team can consume. Teams move faster because they're not rebuilding
  buttons; they're building features.
</p>

<div className="ph-callout">
  <strong>The two-hat rule:</strong> Any engineer on the team should be able to put on a
  designer hat and produce a screen that feels native to the product — and vice versa. The
  design system is the shared vocabulary that makes this possible.
</div>

</div>

## Token-first design

### What are tokens?

<p>
  Design tokens are named, versioned design decisions — expressed as CSS custom properties.
  They are the bridge between Figma and code. Instead of hardcoding <code>#0378a7</code>
  everywhere, teams use <code>--color-user-base</code>, which can be updated in one place
  and propagated everywhere instantly.
</p>

### Three layers

<p>
  Tokens are organised in three layers of abstraction:
</p>

| Layer | Example | Purpose |
|-------|---------|---------|
| **Primitives** | `--color-blue-500` | Raw values — never used in components directly |
| **Semantic** | `--color-user-base` | Intent-named aliases that reference primitives |
| **Component** | `--button-primary-bg` | Scoped to a single component; references semantic tokens |

<p>
  This structure means you can retheme an entire product by changing only the semantic
  layer — every component updates automatically.
</p>

</div>

## API design guidelines

<p>
  Every component API follows these conventions so the design system stays learnable as it grows.
</p>

### Inputs follow Figma

<p>
  If a Figma component has a property called <code>type</code> with options
  <code>primary | secondary | error</code>, the Angular component has an
  <code>@Input() type</code> with the same options. Teams reading the Figma spec and the
  code spec should never have to mentally translate.
</p>

### Boolean inputs are positive

<div className="ph-two-col">
  <div className="ph-do">
    <div className="ph-do-label">✓ Do</div>
    <ul>
      <li><code>[disabled]="true"</code></li>
      <li><code>[iconOnly]="true"</code></li>
      <li><code>[alertIndicator]="true"</code></li>
    </ul>
  </div>
  <div className="ph-dont">
    <div className="ph-dont-label">✗ Don't</div>
    <ul>
      <li><code>[notEnabled]="true"</code></li>
      <li><code>[hideLabel]="true"</code></li>
      <li><code>[noAlert]="false"</code></li>
    </ul>
  </div>
</div>

### Events are past-tense verbs

<p>
  Output event names describe something that <em>has happened</em>:
  <code>buttonClick</code>, <code>dividerClick</code>, <code>chipSelected</code>.
  Consumers subscribe to state changes, not imperative commands.
</p>

### No layout in components

<p>
  Components control only their own internal spacing and sizing. They never set
  <code>margin</code> on <code>:host</code>. Layout is always the consumer's responsibility.
  This keeps components composable in any grid or flex context.
</p>

</div>

<div className="ph-section">

## Figma ↔ code parity

<p>
  Every component in the library maps to a node in the Figma CODE-A-TON Library file.
  When Figma variables change, a CI job syncs the updated token values into
  <code>_primitives.scss</code> and <code>_semantic.scss</code>. Component-level tokens
  in <code>_component.scss</code> are maintained manually to capture Figma-specific overrides
  (shadow, gradient borders, AI chip gradients, etc.).
</p>

<div className="ph-callout">
  <strong>Workflow:</strong> Design changes a token in Figma → CI sync opens a PR →
  engineers review token drift → merge → Chromatic catches visual regressions.
  No manual hex hunting required.
</div>
