## Token architecture

<p>
  The design system uses a three-layer token architecture. Each layer references
  the one above it — consumers should always use the highest-abstraction token that
  fits their use case.
</p>

<table className="fn-table">
  <thead>
    <tr>
      <th>Layer</th>
      <th>File</th>
      <th>Example</th>
      <th>Use in</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>Primitives</strong></td>
      <td><code>_primitives.scss</code></td>
      <td><code>--color-blue-500: #0378a7</code></td>
      <td>Semantic layer only</td>
    </tr>
    <tr>
      <td><strong>Semantic</strong></td>
      <td><code>_semantic.scss</code></td>
      <td><code>--color-user-base: var(--color-blue-500)</code></td>
      <td>Product code, component tokens</td>
    </tr>
    <tr>
      <td><strong>Component</strong></td>
      <td><code>_component.scss</code></td>
      <td><code>--button-primary-bg: var(--color-user-50)</code></td>
      <td>Component SCSS only</td>
    </tr>
  </tbody>
</table>

<div className="fn-callout">
  <strong>Rule:</strong> Never use a primitive token (e.g. <code>--color-blue-500</code>)
  directly in a component or product stylesheet. Always use semantic or component tokens.
  This ensures a future theme swap touches only the semantic layer.
</div>

## Motion and animation

<p>
  Transitions follow a consistent duration and easing to create a cohesive feel.
  Prefer CSS transitions over JavaScript-driven animation where possible.
</p>

<table className="fn-table">
  <thead>
    <tr><th>Token</th><th>Value</th><th>Use</th></tr>
  </thead>
  <tbody>
    <tr><td><code>--duration-fast</code></td><td>120ms</td><td>Hover states, focus rings</td></tr>
    <tr><td><code>--duration-base</code></td><td>200ms</td><td>Dropdowns, tooltips</td></tr>
    <tr><td><code>--duration-slow</code></td><td>300ms</td><td>Modal enter/exit, page transitions</td></tr>

  <strong>Reduced motion:</strong> Always wrap transitions in a
